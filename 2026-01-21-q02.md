\# 2026-01-21 / Q02 — 멀티 리전 DR(최소 다운타임) 설계



\## 1) 문제를 쉬운 말로 재해석

* 지금 서비스는 ALB + Auto Scaling + EC2 로 운영 중이다. 데이터는 DynamoDB를 사용한다.
* 한 리전에서 장애가 나면 다른 리전에서 바로 서비스가 될 수 있도록 하고 싶다.
* 다운타임을 최소화 하는것이 목표이다.



\## 2) 알게된 개념

* &nbsp;Auto Scaling 그룹 - 트래픽이나 부하 변화에 따라 EC2 인스턴스를 자동으로 늘리거나 줄여주는 논리적인 그룹
* DynamoDB 전역 테이블로 구성 - 멀티리전 자동 복제 (읽기/쓰기 모두 가능)
* Route 53 장애조치 - 기본 서비스가 죽으면 자동으로 대체 리전/서버로 트래픽을 넘겨주는 DNS 라우팅 방식 



\## 3) 선택지 하나씩 제거 (핵심만)

* B/C (CloudFormation으로 "필요할 때 생성"):
* 장애 발생 후에 생성하면 시간이 발생 → 다운타임 증가
* \- D (CloudWatch 알람 → Lambda로 Route53 업데이트):
* &nbsp; - 가능은 하지만 Route 53 장애조치 자체가 더 단순/표준
* &nbsp; - 굳이 Lambda로 수동 업데이트 흐름을 만들 이유가 약함
* \- A: 
* &nbsp; DR 리전에 ASG+LB를 미리 만들어 둠 → 즉시 받기 가능
* &nbsp; DynamoDB를 Global Table로 구성 → 멀티 리전 데이터 동기화
